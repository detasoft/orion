# S3 Pack Storage Client

## Goal

Add an efficient Orion-owned S3-compatible storage client for MinIO and AWS S3
that can store and read uploaded Git pack files.

The client should be a storage primitive for the Git pack parser and builder
work. Uploaded packs should be persisted as binary objects and then read back as
streams or bounded byte ranges without routing through JGit-specific storage
hooks.

## Scope

Support a focused pack-storage API:

- put a pack from an `InputStream` without loading the full pack into memory;
- get a pack as an `InputStream`;
- read byte ranges for pack parsing, validation, and future index access;
- check existence and object metadata;
- list packs by repository and prefix;
- delete incomplete or obsolete pack objects;
- store pack hash, object count, source, creation time, and validation status as
  metadata or sidecar records.

The implementation should support both AWS S3 and MinIO through the same Orion
interface. Provider-specific code should stay behind the client boundary.

## Design Notes

Store pack bytes directly. Do not base64-encode pack chunks and do not split the
pack into ad hoc text records unless multipart upload requires backend-managed
parts.

Use streaming upload for small packs and multipart upload for larger packs.
Multipart writes must handle abort-on-failure so partially uploaded packs do not
look valid to readers.

Use a stable key layout that can support multiple repositories and pack
generations, for example:

- `repositories/<repo>/packs/<pack-hash>.pack`
- `repositories/<repo>/packs/<pack-hash>.meta`
- `repositories/<repo>/tmp/<upload-id>.pack`

Writes should be staged under a temporary key first, validated, then promoted to
the final pack key. Readers should only see validated pack objects.

Make checksum validation explicit. At minimum, compare the Git pack trailer hash
with the key or metadata before marking a pack valid. Also record S3 ETag or
provider checksum fields when available, but do not treat them as Git pack
identity.

The client should expose retry and timeout configuration, but keep retry policy
bounded. Failed reads and writes must include bucket, key, operation, and whether
the failure is retryable.

## Integration With Pack Work

The pack parser should be able to consume a stream or range reader from this
client. The pack builder should be able to stream its generated bytes directly
to this client.

Once the Orion parser and builder are available, the S3 client should become the
common storage layer for uploaded receive-pack data and generated upload-pack
responses.

## Verification

Cover at least these cases:

- upload and read back a small binary pack byte-for-byte;
- upload and read back a multipart-sized pack byte-for-byte;
- range reads from the beginning, middle, end, and beyond the end of a pack;
- checksum mismatch keeps an uploaded pack out of the visible final namespace;
- failed multipart upload aborts temporary backend state;
- list returns only validated packs for the selected repository prefix;
- the same API works against MinIO and AWS S3-compatible client tests;
- pack parser can parse a pack read from the S3 client without loading it fully
  into memory.
