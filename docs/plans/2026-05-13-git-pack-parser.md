# Git Pack Parser and Builder

## Goal

Move Orion away from JGit for Git pack-file handling by adding Orion-owned pack
parsing and pack building that can reproduce JGit-compatible packs byte for
byte under fully controlled inputs.

Orion should be able to inspect pack stream contents without depending on a full
repository import path, then assemble valid pack files for fetch and
receive-pack flows. JGit remains useful as a reference implementation while the
Orion implementation is being built and verified. The target verification mode
is byte-for-byte equality between Orion-built and JGit-built packs when all
inputs and generation parameters are fixed.

## Scope

Support reading the core Git pack format:

- pack header validation, including magic, version, and object count;
- object entry headers and variable-length size decoding;
- whole-object entries for commit, tree, blob, and tag objects;
- `OFS_DELTA` and `REF_DELTA` entries as parsed delta records;
- trailing pack checksum verification.

Support writing valid pack files from Orion object data:

- pack header generation;
- whole-object entry encoding;
- delta entry encoding when a delta source is provided;
- deterministic object ordering;
- configurable compression settings matching the JGit reference run;
- configurable delta selection policy matching the JGit reference run;
- trailing pack checksum generation.

Delta resolution can be a separate step from raw pack parsing. The parser should
expose enough structured information for an object reconstruction component to
resolve base objects and apply deltas. The builder should accept already
resolved objects first, then grow support for emitting deltas after the raw
format is stable.

## Design Notes

Keep the parser and builder independent from storage backends. S3, filesystem,
and test fixtures should be able to feed the same byte stream into the parser
and consume the same generated pack stream from the builder.

Prefer a streaming API that does not require loading large packs fully into
memory. Small test fixtures can still use in-memory helpers for readability.

Parser errors should include the byte offset and the violated pack invariant
where possible. Invalid or truncated packs must fail before any storage layer is
asked to persist parsed objects.

Byte-for-byte equality with JGit is the desired regression target. Tests should
build fixtures where every Git-object input is explicit and stable:

- blob contents;
- tree entries and entry ordering;
- commit tree, parents, author name/email/time/timezone, committer
  name/email/time/timezone, encoding, message, and extra headers;
- tag object target, type, tagger name/email/time/timezone, message, and extra
  headers.

Tests must also control every pack-generation choice that can change bytes:

- object traversal and output order;
- delta enabled/disabled state;
- delta base choice and delta byte encoding;
- `OFS_DELTA` versus `REF_DELTA` preference;
- zlib compression level, strategy, nowrap behavior, and implementation
  compatibility;
- pack index or checksum inputs where relevant.

Canonical pack equality is still useful as a diagnostic layer when byte equality
fails. After parsing both packs, Orion and JGit should agree on object IDs,
object types, object contents, delta base references, and pack checksum
validity, but a successful implementation should make the deterministic
fixtures pass with identical pack bytes and therefore identical pack hashes.

## Verification

Cover at least these cases:

- parse an empty valid pack generated by JGit;
- parse JGit-generated packs with whole commit, tree, blob, and tag objects;
- parse JGit-generated packs containing `OFS_DELTA` and `REF_DELTA` records;
- build Orion packs for the same object sets and verify JGit can parse them;
- compare Orion-parsed and JGit-parsed pack contents through a canonical model;
- compare Orion-built and JGit-built packs byte-for-byte for deterministic
  no-delta fixtures;
- compare Orion-built and JGit-built packs byte-for-byte for deterministic delta
  fixtures after delta generation is implemented;
- invalid magic, unsupported version, truncated object body, and checksum
  mismatch;
- parsing the same fixture through more than one backend-facing input source.
